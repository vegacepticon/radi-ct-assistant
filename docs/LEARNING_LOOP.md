# RadiCT Assistant — learning loop

Документ описывает целевой рабочий процесс, в котором RadiCT Assistant постепенно адаптируется к стилю Романа на основе реальных правок: черновик → исправление → объяснение → сохранение примера/правила → более точные следующие ответы.

## Цель

Система должна помогать формировать:

1. **Заключения** по готовой описательной части РКТ/КТ.
2. **Описательную часть** из произвольных фраз, черновых находок или расшифровки голоса.
3. **Описание + заключение** из неструктурированного набора находок.
4. **Исправленные обучающие примеры** для few-shot retrieval.
5. **Кандидаты правил стиля**, которые затем можно переносить в Hermes skill `ct-protocol-drafting-roman-style`.

Главный принцип: система не должна молча “обучаться” на любом медицинском тексте. Сохранение клинического случая в базу примеров выполняется только после явной команды пользователя и проверки на PHI.

---

## Базовый цикл

```text
Ввод Романа
  ↓
Draft: ассистент формирует черновик описания/заключения
  ↓
Review: Роман принимает, редактирует или отклоняет
  ↓
Feedback: Роман объясняет, почему его вариант лучше
  ↓
Persist:
  - сохранить пример описание → финальное заключение;
  - сохранить исправление и ошибку ассистента;
  - сформировать кандидат правила;
  - при необходимости обновить style skill.
  ↓
Retrieval: следующие похожие случаи подтягивают сохраненный пример
```

## Основные сценарии

### 1. Заключение из готового описания

Пользовательский запрос:

```text
РКТ заключение.
Анамнез: рак прямой кишки.
Сравнение: 27.04.2026.
Описание:
...
```

Ожидаемый ответ: только блок `Заключение`, без преамбулы и без повторения описания.

Система должна:

- определить основную клиническую задачу;
- выделить динамику;
- сгруппировать находки по клинической значимости;
- не добавлять факты, которых нет в описании;
- не пропускать релевантные стабильные, новые и разрешившиеся находки.

### 2. Заключение с предыдущим протоколом

Пользовательский запрос:

```text
Нужна помощь с заключением.
Предыдущий протокол:
...

Свежее описание:
...
```

Система должна:

- сравнить свежее описание с предыдущим протоколом;
- сформировать вывод о положительной, отрицательной или смешанной динамике;
- отдельно указать стабильные релевантные находки;
- отдельно указать новые и разрешившиеся находки, если они клинически значимы;
- не использовать формальную терминологию RECIST без явного расчета и выбора таргетных очагов.

### 3. Описание из произвольных фраз

Пользовательский запрос:

```text
РКТ описание.
Область: ОГК.
Сравнение: есть.
Черновые находки:
S8 справа очаг был 23 на 23, стал 20 на 13 на 17.
S1 субсолидный 5.5 без динамики.
Прекардиальные были 13, стали 8.
Плевра свободна.
```

Ожидаемый ответ: структурированная описательная часть в стиле протокола.

Система должна:

- нормализовать диктовку и неполные фразы;
- сохранить смысл, локализацию, размеры и динамику;
- не достраивать отсутствующие данные;
- при необходимости задавать короткие уточняющие вопросы.

### 4. Описание + заключение из произвольных фраз

Пользовательский запрос:

```text
РКТ описание + заключение.
Анамнез: ...
Сравнение: ...
Черновые находки:
...
```

Система сначала формирует описание, затем заключение на его основе.

Важно: заключение должно вытекать из сформированного описания. Если описание было изменено пользователем, заключение надо строить по финальной версии описания, а не по исходным черновым фразам.

### 5. Голосовой ввод

Целевой workflow:

```text
Голосовое сообщение
  ↓
Расшифровка
  ↓
Очистка от мусора диктовки и потенциальных PHI
  ↓
Структурирование находок
  ↓
Описание / заключение
  ↓
Исправление Романа
  ↓
Сохранение примера или правила
```

Ограничение: реальные персональные и медицинские идентификаторы пациента не должны отправляться во внешние LLM/API. Для сохранения в базу нужны анонимизированные тексты.

### 6. Исправление варианта ассистента

Пользовательский запрос:

```text
Исправляю:
...

Почему:
1. ...
2. ...

Сохрани как пример.
```

Система должна:

- сохранить финальный вариант Романа как authoritative target;
- сохранить черновик ассистента как rejected/previous draft;
- сохранить объяснение как feedback;
- извлечь теги ошибок;
- предложить или выполнить сохранение примера, если пользователь явно попросил;
- предложить обновить skill, если исправление является обобщаемым правилом.

## Команды пользователя в Telegram

Рекомендуемые короткие инициаторы:

| Команда | Назначение |
|---|---|
| `РКТ заключение` | Сформировать заключение по описанию |
| `РКТ описание` | Сформировать описательную часть из черновых находок |
| `РКТ описание + заключение` | Сформировать оба блока |
| `Исправляю:` | Передать финальную правку |
| `Почему:` | Объяснить, почему правка лучше |
| `Сохрани как пример` | Явно разрешить сохранить анонимизированный случай в базу few-shot |
| `Это правило на будущее` | Явно указать, что правка должна стать правилом стиля |

## Слои хранения

### 1. Hermes skill

Используется для устойчивых правил стиля.

Примеры:

- не писать `РКТ-признаки`, если модальность уже очевидна из протокола;
- не использовать `без существенной динамики`;
- в сравнительных онкологических исследованиях учитывать не только уменьшение опухолевых очагов, но и разрешившиеся сопутствующие изменения;
- во втором абзаце заключения перечислять релевантные находки без динамики, а не заменять их общей фразой.

### 2. Reference base

Папка для качественных анонимизированных примеров, используемых few-shot retrieval.

```text
data/references/clean/
```

Каждый пример — пара:

```text
Описание → финальное заключение Романа
```

### 3. Case store

Папка для рабочих случаев и истории правок.

```text
data/cases/
  drafts/
  accepted/
  corrected/
```

### 4. Feedback log

Журнал правок и объяснений.

```text
data/feedback/feedback_log.jsonl
```

Каждая строка — отдельная JSON-запись feedback event.

### 5. Lesson candidates

Кандидаты правил, которые еще не перенесены в skill.

```text
data/feedback/lesson_candidates/
```

## Формат reference-файла

Файлы в reference base должны быть анонимизированы и иметь минимальный YAML frontmatter.

```yaml
---
анамнез: рак прямой кишки
область:
  - ОГК
  - ОБП
  - ОМТ
сравнение: true
экстренность: false
статус: true
задача: conclusion
динамика: положительная
---

Описание:
...

Заключение:
...
```

Обязательные безопасные поля:

| Поле | Назначение |
|---|---|
| `анамнез` | краткий обезличенный клинический контекст |
| `область` | область исследования, список строк |
| `сравнение` | есть ли динамическое сравнение |
| `экстренность` | экстренное исследование или нет |
| `статус` | `true` — включать в few-shot |

Расширенные поля, допустимые после проверки:

| Поле | Пример |
|---|---|
| `задача` | `conclusion`, `description`, `description_and_conclusion` |
| `динамика` | `положительная`, `отрицательная`, `смешанная`, `без динамики` |
| `модальность` | `КТ`, `РКТ`, если нужно для фильтрации, но не для текста заключения |
| `онкология` | `true` |

Запрещено хранить:

- ФИО;
- ID пациента/исследования;
- дату рождения;
- учреждение;
- врача;
- точные даты исследования, если они могут идентифицировать случай;
- номера телефонов, адреса, полисы, любые прямые идентификаторы.

## Input

...

## API roadmap

Текущий endpoint `/api/generate` остается для совместимости, но для learning loop нужны отдельные операции.

### `POST /api/draft`

Создать черновик и вернуть `case_id`.

Вход:

```json
{
  "task": "conclusion",
  "input_text": "...",
  "area": ["ОГК", "ОБП", "ОМТ"],
  "clinical_context": "рак прямой кишки",
  "comparison": true,
  "mode": "fast"
}
```

Выход:

```json
{
  "case_id": "2026-07-06-001",
  "draft": "...",
  "references_used": ["..."]
}
```

### `POST /api/accept`

Пользователь принял черновик без правок.

```json
{
  "case_id": "2026-07-06-001",
  "save_as_reference": true
}
```

### `POST /api/correct`

Пользователь исправил черновик.

```json
{
  "case_id": "2026-07-06-001",
  "roman_final": "...",
  "feedback": "...",
  "save_as_reference": true,
  "create_lesson_candidate": true
}
```

### `POST /api/references/promote`

Перевести исправленный case в few-shot reference base после PHI-проверки.

### `GET /api/lessons`

Показать кандидаты правил.

## CLI roadmap

CLI нужен раньше API, потому что его проще тестировать локально.

```bash
python scripts/radi_ct.py draft input.md --task conclusion
python scripts/radi_ct.py accept CASE_ID --save-as-reference
python scripts/radi_ct.py correct CASE_ID --final final.md --feedback feedback.md --save-as-reference
python scripts/radi_ct.py lessons
python scripts/radi_ct.py promote CASE_ID
python scripts/radi_ct.py index
```

## Prompt modes

Нужно отделить режимы генерации, потому что “заключение из описания” и “описание из черновых фраз” — разные задачи.

| Mode | Вход | Выход |
|---|---|---|
| `conclusion_from_description` | готовое описание | заключение |
| `description_from_notes` | черновые фразы/диктовка | описание |
| `description_and_conclusion_from_notes` | черновые фразы/диктовка | описание + заключение |
| `edit_existing_description` | существующее описание + пожелания | исправленное описание |
| `edit_existing_conclusion` | существующее заключение + пожелания | исправленное заключение |

Для каждого режима нужен отдельный system prompt в `prompts/`.

## PHI safety rules

1. Система не сохраняет случай в reference base без явной команды пользователя.
2. Перед сохранением надо проверить YAML и тело текста на прямые идентификаторы.
3. Whitelist YAML-полей предпочтительнее blacklist.
4. Тело медицинского текста не должно автоматически редактироваться PHI scrubber без отдельной логики, потому что можно повредить медицинский смысл.
5. Если есть сомнение, сохранить case как draft/corrected локально нельзя без подтверждения пользователя.
6. Реальные patient data не должны отправляться во внешние LLM/API.

---

## Правила promotion в reference base

Случай можно сделать few-shot примером, если:

- есть финальный вариант Романа;
- пример обезличен;
- текст отражает предпочтительный стиль;
- нет грубых опечаток и временных черновых фраз;
- метаданные достаточно заполнены для retrieval;
- пользователь явно разрешил сохранение.

Не сохранять как reference, если:

- случай слишком частный и вводит неверное правило;
- есть PHI;
- финальный вариант не проверен;
- пользователь просто обсуждал формулировку, но не одобрил финал;
- текст содержит реальные идентификаторы пациента или учреждения.

---

## Правила promotion в skill

Feedback стоит переносить в skill, если он обобщаемый.

Примеры обобщаемых правил:

- “Не писать `РКТ-признаки`, если модальность очевидна”.
- “В сравнительном заключении указывать разрешившиеся изменения, если они описаны”.
- “Стабильные релевантные находки лучше перечислять, а не заменять общей фразой”.

Не переносить в skill:

- одноразовые решения конкретного случая;
- диагностические выводы, завязанные на конкретные изображения;
- временные предпочтения;
- противоречивые правки без подтверждения.

## Пример текущего learning event

Исходная ошибка ассистента:

- использовал избыточную приставку `РКТ-признаки`;
- не указал разрешение линейных уплотнений в средней доле правого легкого;
- не перечислил все релевантные находки без динамики во втором абзаце.

Финальное правило:

```text
В сравнительных онкологических исследованиях заключение должно указывать не только уменьшение размеров опухолевых проявлений и лимфоузлов, но и разрешившиеся релевантные находки. Если модальность очевидна из заголовка протокола, не использовать `РКТ-признаки`. В отдельном стабильном абзаце перечислять релевантные находки без динамики, если они явно описаны.
```

Статус: правило уже перенесено в Hermes skill `ct-protocol-drafting-roman-style`.
